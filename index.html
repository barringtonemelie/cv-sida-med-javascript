explain it to me like i'm five - the World Wide Web

Vem har inte känt någon gång, när man stöter på ett nytt koncept, att man önskar att någon kunde bryta ner det i små, hanterbara delar, eller förklara det som att man vore fem år gammal? I den här artikeln ska jag försöka ge mig på att göra just det - att förklara webben och dess tekniker, hur de hänger ihop, och vad tillgänglighet och prestanda har med det hela att göra.

Om du läser den här artikeln använder du antagligen någon form utav dator eller mobil. Helt säkert är i alla fall att du använder dig av en webbläsare, kanske Google Chrome, Safari, eller Edge. Den här webbläsaren är ett program som läser in kod, eller instruktioner skrivna i ett språk som webbläsaren och därefter datorn kan förstå, och sedan utför de instruktionerna. En webbläsare läser raderna med kod uppifrån och ner, med några få undantag. De allra vanligaste språken som används för att skriva instruktionerna kallas för HTML, CSS och JavaScript. En hemsida består vanligtvis av flera separata filer, där en fil innehåller ett programmeringsspråk åt gången.

Vad består då egentligen Internet av, rent fysiskt? Man kan grundläggande säga att det består utav olika enheter som pratar med varandra, eller skickar information fram och tillbaka. Enheterna kan vara mobiler, datorer, plattor, TV-apparater, smarta enheter i hemmet mm. Kommunikationen sker på kort distans trådlöst genom radiovågor, t.ex. via WiFi eller det mobila nätet. Men det sker över hela världen via fysiska kablar som sammanbinder alla dessa enheter. Det som sedan möjliggör att alla dessa enheter kan gå in på olika hemsidor och hämta hem information är servrar - egentligen också datorer, men deras enda syfte är att betjäna andra enheter och bl.a. tillhandahålla information. Det finns enorma mängder servrar över hela världen på olika platser.

När det kommer till språken som bygger upp internet, är det allra viktigaste språket HTML. Som språk beskrivs det ofta som själva skelettet som bygger upp hemsidan och ger den dess struktur. Det är ett totalt nödvändigt språk för att skapa hemsidor idag. Det är HTML som man som utvecklare sedan utgår ifrån och refererar till för att ge hemsidan ett utseende och en funktionalitet. Det är också framförallt via HTML som man lägger in sidans innehåll såsom text och bild, och det märker också upp vad som är en rubrik, vad som är en paragraf, bild eller länk, etc. Därför kallas det också för ett “markup language”. Delarna eller bitarna av sidan i HTML, eller för att anspela på liknelsen med ett skelett, “benen”, kallas för element.

CSS är språket som man skriver instruktionerna på för hemsidans form och färg. Med CSS kan man ge texten och bakgrunden färg, och man kan skapa utseende och form på elementen på sidan. Det går också att skapa enklare animationer och effekter.

JavaScript använder man för att ge hemsidan dess funktionalitet. Det är ett språk som började väldigt litet, men har vuxit enormt och är idag ett omfattande och varierande språk som kan användas för mycket. Med JavaScript kan man skapa och ta bort element, man kan ändra dess stil, och mycket mer, allt genom att referera och spara hänvisningar till elementen i JavaScript-filen. Av de tre nämnda språken, så är JavaScript det språk som liknar ett mer “klassiskt” programmeringsspråk, dvs. det människor kanske föreställer sig när de tänker sig hur programmering ser ut som de sett på film, när hackern frenetiskt skriver kod för att sedan avsluta med den dramatiska repliken: “I’m in”.

För många programmerare är JavaScript ett antingen hatat eller älskat språk. Man kan i alla fall konstatera att det har sina egenheter, vilket kanske har att göra med att det började med att bara vara små korta programrader, eller skript, men idag skrivs stora program och applikationer i JavaScript.

Men hur knyts då alla dessa delar samman? Hur gör webbläsaren för att tolka allting? Som jag nämnde innan, så läser den koden rad för rad. När den har fått filerna för en hemsida skickade till sig av en server, via något som kallas HTTP-protokollet, börjar den med att leta efter en fil som heter index.html. Det är denna fil som sedan kommer guida webbläsaren till övriga resurser, som CSS-filen och JavaScript-filen. Dessa är nämligen länkade i index.html. Hur webbläsaren sedan visar upp sidan för dig som användare styrs av specifikationer underhållna av W3C (World Wide Web Consortium) men det kan fortfarande finnas vissa små skillnader webbläsare emellan, i t.ex. hur saker ser ut.

Både HTML och CSS läses rad för rad, och det gör även JavaScript, men det finns några få undantag. I exempelvis JavaScript, läser webbläsaren in namn på funktionerna först (namngivna, avgränsade block utav kod) vilket gör att man kan anropa (be webbläsaren att köra) en funktion innan den egentligen är skriven. Det är en trevlig liten funktionalitet för oss som utvecklare, eftersom det kan göra koden väldigt översiktlig och lättläst om man är duktig på att ge sina funktioner beskrivande namn. Det underlättar också underhåll utav koden.

Inom webbutveckling brukar man dela upp webben i två delar - frontend och backend. Frontend är det användaren ser och kan interagera med, medan backend är det som sker bakom kulisserna. T.ex. det jag nämnde innan, att en användare ber att få se en hemsida genom att klicka på en länk. Då skickas en förfrågan iväg först till servern som håller reda på alla adresser på internet, DNS-servern. (DNS står för Domain Name Server och man kan likna den vid internets telefonkatalog.) Den i sin tur guidar användaren vidare till servern där den efterfrågade hemsidans filer ligger sparade. Servern skickar i sin tur tillbaka filerna till användarens webbläsare, som renderar informationen till webbsidan som användaren kan se och interagera med, och därmed har vi rest från frontend, till backend, och tillbaka till frontend igen. Den här kommunikationen sker enligt ett protokoll kallat för HTTP - det styr reglerna för hur den här kommunikationen ska gå till, vem som ska svara när osv.

Allt detta är ju väldigt intressant och trevligt, och otroligt användbart. Men säg att en användare är synskadad. Plötsligt kanske den där snygga och flashiga sidan inte alls är lika användbar längre, och information riskerar att gå förlorad för den användaren. Därför är det väldigt viktigt att vi som utvecklare ser till att skriva koden till våra webbsidor på ett sätt som är enkelt för hjälpmedlen som t.ex. en synskadad person skulle använda. Ta en skärmläsare som exempel. Den kommer med hjälp av hur innehållet är märkt upp i HTML-koden, hantera innehåll och text som antingen rubriker, meny, brödtext, bilder, mm. på det sätt som blir mest logiskt för användaren. För att det ska funka behöver utvecklare vara noggranna när de märker upp innehållet i sin HTML, så att koden verkligen beskriver vad innehållet är, och i vilken ordning det bör läsas upp av skärmläsaren.

Det kan också röra sig om andra funktionsnedsättningar, det kan t.ex, handla om användare som bara kan använda sig av ett tangentbord och inte en datormus. Då är det värdefullt att ordningen som användaren kan tabba sig runt på hemsidan är logisk. Allt detta och många fler sätt att skriva tillgängliga hemsidor på finns specificerade på webbriktlinjer.se.

Ytterligare en sak som utvecklare behöver ta med i beräkningen är prestanda. Hur man skrivit sin kod kan påverka hur snabb och smidig hemsidan upplevs av användaren. I mindre sammanhang, t.ex. om en hemsida inte har särskilt många besökare, är det inte alltför svårt att se till att hemsidan och dess innehåll flyter på snabbt och smidigt. Det kan handla om att anpassa storleken på bilder, så att de har liten filstorlek och kan laddas in snabbt.

Men om det är en hemsida som börjar få många användare, kanske flera tusen eller miljoner användare, så blir det här mycket viktigare. Då kan små förändringar i hur man väljer att bearbeta informationen på sin hemsida via JavaScript exempelvis ha stor effekt för hur den enskilda användaren upplever hemsidan. Detta beror på att servern i sammanhang med många användare kan få många anrop samtidigt, då ska samma kod i backend köras massvis med gånger, och då tar det plötsligt tid för servern att svara. Därför behöver man som utvecklare testa hur snabb ens kod egentligen är.

Det finns också saker man kan göra som utvecklare för att detta inte ska upplevas lika störande av ens användare. T.ex. att om en viss del av hemsidan riskerar att behöva lång tid på sig för att laddas, så kan man ge användaren information att något händer, att sidan håller på att laddas, genom exempelvis informerande text eller en liten cirkel som snurrar.

Jag hoppas att jag lyckades med att bryta ner några grundläggande koncept på webben åt dig. Tack för att du läste!
